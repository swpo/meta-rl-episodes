# Verifiers Environment Patterns

This note distills Verifiers patterns to use when building small meta RL
environments. Keep it portable: a clean checkout of this repo should contain
the context needed to reproduce the setup.

## Entrypoint

Expose a `load_environment(...) -> vf.Environment` function. Accept `**kwargs`
for forward compatibility with Prime and local test harnesses.

For new synthetic envs, keep pure helpers importable without `verifiers` when
possible:

- dataset/example generation
- parsing
- reward scoring
- malformed-output handling

Then import `datasets` and `verifiers` lazily inside `load_environment`. This
lets local unit tests run even when the full Verifiers runtime is unavailable.

## Dataset Rows

Rows should include:

- `prompt`: chat messages for the initial user prompt
- `answer`: compact ground-truth payload, often JSON
- `task`: stable task family string
- `info`: structured metadata used by the env and reward

For multi-turn envs, put follow-up user messages and per-turn ground truths in
`info`.

## Multi-Turn Env Shape

Use `vf.MultiTurnEnv` for environments that ask the model to respond across
multiple user turns.

Common pattern:

```python
class MyEnv(vf.MultiTurnEnv):
    @vf.stop
    async def max_turns_for_example(self, state: vf.State) -> bool:
        return len(state["trajectory"]) >= state["info"]["num_turns"]

    async def env_response(self, messages: vf.Messages, state: vf.State, **kwargs):
        assistant_count = len([m for m in messages if m["role"] == "assistant"])
        follow_up_idx = assistant_count - 1
        return [vf.UserMessage(content=state["info"]["follow_ups"][follow_up_idx])]
```

`state["trajectory"]` counts model responses, so this stop condition ends after
the intended number of assistant turns.

## Reward Shape

Use defensive reward functions:

```python
def reward(completion, answer, **kwargs) -> float:
    ...
```

Then attach them with:

```python
rubric = vf.Rubric()
rubric.add_reward_func(reward, weight=1.0)
```

Reward functions should return `0.0` or shaped partial credit for routine model
mistakes such as missing tags, invalid JSON, empty content, `None`, missing tool
calls, or malformed tool outputs. They should not raise on ordinary bad
completions.

Hosted Verifiers scoring passes message objects during reward computation, even
though stored rollout samples are serialized to dicts. Tests should cover both:

```python
def message_content(message):
    if isinstance(message, dict):
        return message.get("content")
    return getattr(message, "content", None)
```

If a local helper also accepts a state-shaped fixture for convenience, add a
separate test that scores the hosted-style `answer` payload directly.

Avoid rewarding "best hidden answer among many" unless the task explicitly asks
for search over candidates. If a parser accepts multiple candidate answers, add
tests and metrics for candidate count, exact-one-answer rate, output length,
code fences or wrappers, and truncation. Prefer scoring a single selected answer
and recording auxiliary metrics over silently taking the maximum score across a
stuffed response.

Attach diagnostic metrics with `rubric.add_metric(...)` when they help explain
where reward comes from. For example, track parseability, exact format
adherence, multi-candidate rate, code-fence rate, and output length separately
from the optimized reward.

## Local Checks

At minimum, test:

- deterministic example generation
- exact-answer reward
- malformed output returns a score rather than raising
- partial-credit behavior
- multi-turn completion averaging or final-state scoring
- hosted-style `completion, answer` reward signature
- dict-shaped messages and object-style Verifiers messages
- parser loopholes such as repeated answers, candidate stuffing, wrappers, or
  long outputs when those could affect reward

Use `PYTHONDONTWRITEBYTECODE=1` when running tests so verification does not
leave generated bytecode in episode directories.

## Status Semantics

Use `episode.yaml` `status: "draft"` and registry status `local-ready` when:

- local deterministic helpers pass
- the env has not yet been pushed, installed, evaled, or trained on Prime
- hosted integration is still unknown

Use `status: "complete"` only when the intended integration path has been
checked, or when the episode is explicitly scoped as local-only.
